To make your SystemC analysis with Claude highly efficient, you need to eliminate repetitive prompting and work around Claude's memory limitations (context window tokens). Large SystemC projects with multiple .cpp and .hpp files can quickly overwhelm a chat session. [1, 2] 
Here is a blueprint to optimize your workflow using your existing GitHub repo and Claude Project.
------------------------------
## 1. Automate Code Feeding (The GitHub Action Method)
Instead of copying and pasting individual files, combine your codebase into a single, structured markdown file that Claude can digest instantly. [3] 
You can automate this by adding a GitHub Action to your repository. Every time you push code, it will automatically update a single codebase.md file that you can drop straight into your Claude Project knowledge base. [4, 5, 6, 7] 
Create a file in your repo at .github/workflows/combine-code.yml:

name: Combine SystemC Codeon: [push]jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Combine files
        run: |
          echo "# SystemC Codebase Snapshot" > codebase.md
          find . -type f \( -name "*.cpp" -o -name "*.hpp" -o -name "*.h" \) ! -path "*/.*" | while read file; do
            echo -e "\n## File: $file\n\`\`\`cpp" >> codebase.md
            cat "$file" >> codebase.md
            echo -e "\n\`\`\`" >> codebase.md
          done      - name: Commit codebase file
        run: |
          git config --global user.name 'Github Bot'
          git config --global user.email 'bot@github.com'
          git add codebase.md
          git commit -am "Update codebase snapshot" || exit 0
          git push


* Why this is efficient: You download just one file (codebase.md) from your GitHub repo and upload it to your Claude Project. It includes all headers and source files cleanly separated by Markdown tags. [8, 9, 10] 

------------------------------
## 2. Set Up Custom Instructions in your Claude Project
Don't waste time telling Claude what SystemC is or explaining your research goals in every new chat. Use the Custom Instructions feature inside your Claude Project settings. [11, 12] 
Paste a template like this into the Project Instructions:

Role: You are an expert hardware modeling engineer specializing in SystemC (IEEE 1666), TLM-2.0, and discrete-event simulation architectures.

Context: I am working on a research project titled [Insert Project Name]. The core goals are [Insert 1-2 sentence goal, e.g., optimizing latency / testing NoC throughput].

Analysis Rules:
1. When evaluating code, always check for SystemC-specific issues: channel synchronization, event/wait race conditions, SC_METHOD vs SC_THREAD misuse, and improper use of sensitive loops.
2. Verify TLM-2.0 compliance (blocking vs non-blocking transport, payload event queues) if applicable.
3. Keep responses highly modular. Suggest structural improvements first, then provide code snippets.
4. If a suggested fix changes multiple modules, clearly map out the structural dependencies.

------------------------------
## 3. Use "Chunked" Query Blueprints
SystemC simulations are prone to hidden bottlenecks (e.g., infinite loops in SC_THREAD without a wait(), or excessive delta-cycle evaluation). To get the best answers, target your questions using specific analysis lenses: [13] 

* For Bottlenecks: "Using the uploaded codebase.md, analyze the execution loops of my SC_THREADs. Are there places where unnecessary delta-cycles are being triggered, or where data caching could speed up the simulation time?"
* For Correctness: "Scan the module interfaces. Are there any race conditions between my sc_signal writes and reads across clock boundaries?"
* For Modularity: "I want to scale this simulation. Is the communication between Module A and Module B tightly coupled? Suggest how to refactor this using TLM-2.0 sockets to improve simulation speed."

------------------------------
## 4. Create an Evaluation Log File
As Claude suggests changes, do not just apply them blindly. Keep an eval_log.md file in your Claude Project knowledge base. [14, 15] 

* Every time Claude successfully solves a bug or improves simulation speed, ask Claude: "Summarize the fix we just made to the SystemC scheduler handling so I can add it to our Project Log."
* Update that text file in your Project. This ensures Claude remembers previous engineering decisions even if you start a completely fresh chat session. [16, 17, 18, 19]
