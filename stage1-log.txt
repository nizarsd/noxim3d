======================================================================
1. DP VS BUFFERLEVEL BEHAVIOUR AND WHEN DP PROVES GOOD
======================================================================

Main observation:
DP does not mainly improve performance in the low-load region. At low PIR, the network is lightly congested, so both DP and bufferlevel selection have similar behaviour. DP becomes useful when the network approaches the congestion knee, where local queue-state selection is no longer sufficient.

Reason:
Bufferlevel is a local heuristic. It chooses based on immediate neighbouring buffer state. This is effective when congestion is weak or spatially shallow, but it can fail near the knee because a locally free direction may still lead into a globally congested region.

DP is a multi-hop cost-to-go metric. It propagates a congestion-aware cost field over the NoC, so the selection unit can avoid directions that are locally acceptable but globally expensive.

Final 5x5x3 full sweep, CINTERVAL = DP_CYCLE:

PIR      BL delay   DP delay   DP delay reduction   DP wins   Throughput change   Energy change
0.016    20.72      18.99      8.35%                3/3       +0.02%              -2.22%
0.018    26.72      21.86      18.18%               3/3       +0.18%              -3.82%
0.020    37.03      25.38      31.46%               3/3       +0.02%              -5.44%
0.025    454.33     97.78      78.48%               3/3       +2.20%              -4.71%
0.030    1535.84    1263.67    17.72%               3/3       +2.40%              -0.36%

Interpretation:
- 0.016-0.020: low-to-moderate congestion. DP advantage increases gradually.
- 0.025: congestion knee. DP gives its strongest benefit.
- 0.030-0.040: deep saturation. DP still wins all seeds, but both policies are overloaded, so the relative improvement shrinks.

Key result:
At PIR = 0.025 for 5x5x3:
BL delay = 454.33
DP delay = 97.78
DP delay reduction = 78.48%

Conclusion for point 1:
DP proves good around the congestion knee. It is less important at low load and less dominant in deep saturation. Its value is strongest when congestion is forming but has not yet overwhelmed the network.

The 5x5x4 case shows lower improvement because the fourth z-layer appears to soften the congestion knee for this configuration. The additional layer gives more spatial spreading and more vertical path diversity, so bufferlevel does not collapse as sharply. Conclusion for point 3:
DP is effective across different NoC configurations, but the magnitude of improvement depends on topology, z dimension, traffic pressure, and where the selected PIR sits relative to the knee. The correct conclusion is not that DP always scales monotonically better with size. The correct conclusion is that DP is a valid multi-hop congestion-aware baseline whose advantage appears when local bufferlevel decisions become insufficient.

======================================================================
4. MILESTONE RESULTS AND SIGNIFICANT FINDINGS
======================================================================

Milestone A - DP implementation corrected:
Several implementation problems in the DP selection unit were identified and corrected:
- rx_dp_cost was uninitialised in dpProcess.
- selectionDP had an unconditional push_back that discarded DP rankings under contention.
- routing_directionsUpdater had reset-ordering and no_dst issues.
- routing_directionsUpdater was changed to clock.pos() for deterministic latching.
- cost_mem[DPSIZE][DIRECTIONS] was added for coherent per-destination Bellman relaxation.
- frozen_local_cost was introduced at pass boundaries.
- A per-hop term was added so cost-to-go includes path length, not congestion only.

Significance:
The DP unit became a valid and testable multi-hop baseline rather than a broken or unstable policy.

Milestone B - Turn-law-aware DP relaxation:
The DP relaxation was made aware of the balanced odd-even turn law through can_turnOddEvenBalanced. This prevents Bellman propagation from considering illegal routing turns.

Significance:
The cost field is now consistent with the actual deadlock-free routing law.

Milestone C - Thermal-management regression resolved:
Disabling the dy_thermal_manage SC_METHOD caused max/min counters to stay zero. Combined with gamma defaulting to 1.0, cost_to_go wrote constant cost values. Thermal management was confirmed inert when dy_t_mode = 0, so the SC_METHOD should not be disabled.

Significance:
This restored congestion awareness in the DP cost computation.

Milestone D - CINTERVAL issue diagnosed:
For small 5x5x3 NoCs, CINTERVAL = DP_CYCLE/4 was too short and produced inconsistent results. Returning to CINTERVAL = DP_CYCLE recovered coherent behaviour and matched earlier clean results.

At PIR = 0.025:
Earlier clean 5x5x3:
BL delay approximately 507.45

Current corrected 5x5x3 with CINTERVAL = DP_CYCLE:
BL delay = 454.33

Significance:
The inconsistency was due mainly to CINTERVAL/statistical sampling effects, not the normalized cost-to-go formulation.

Milestone E - Final 5x5x3 knee result:

Significance:
This is the strongest clean Stage 1 evidence that DP can outperform bufferlevel near the congestion knee.

Milestone F - Buffer-size sensitivity:

Milestone G - NoC-size/configuration comparison:

Significance:
DP improvement is not universal or monotonic with NoC size. It depends on how each configuration enters congestion. This is an important scientific finding because it prevents overclaiming.

======================================================================
FULL STAGE 1 CONCLUSION
======================================================================

Stage 1 can be concluded as a validation and baseline-characterisation stage.

The simulator now includes a corrected deadlock-free balanced odd-even routing scheme and a corrected distributed DP selection unit. The DP unit performs turn-law-aware Bellman-style cost propagation and uses a normalized congestion-aware cost-to-go with a hop component. The most important implementation bugs that previously invalidated DP behaviour have been identified and fixed.

The experimental evidence shows that DP selection is a valid multi-hop congestion-aware baseline. Its strongest advantage appears near the congestion knee, where local bufferlevel selection can make poor decisions because it sees only immediate queue state. Around the knee, DP can substantially reduce average delay by using a propagated cost field that captures downstream congestion.

The final 5x5x3 run with CINTERVAL = DP_CYCLE gives the clearest result:
At PIR = 0.025, bufferlevel delay is 454.33 cycles, while DP delay is 97.78 cycles, corresponding to a 78.48% delay reduction. DP also wins all three seeds at this point.

The 8x8x3 configuration confirms that this behaviour can also appear in a larger NoC. At PIR = 0.016, DP reduces delay from 458.68 to 130.60 cycles, a 71.53% reduction.

The buffer-depth sweep confirms that the DP benefit is not limited to a single buffer size. DP remains better under buffer sizes 8, 16, and 32, although the relative improvement decreases when larger buffers absorb some congestion.

The 5x5x4 result provides an important counterbalance. Although DP still improves performance, the representative knee improvement is only 33.50%. This suggests that the additional z-layer softens the congestion knee and gives bufferlevel more spatial/path diversity, making local selection less fragile in that configuration. Therefore, DP benefit is topology- and knee-dependent.

The correct scientific claim for Stage 1 is therefore:

Corrected DP selection is a valid and effective multi-hop congestion-aware baseline. It can significantly outperform bufferlevel near the congestion knee, but its advantage is not universal, not monotonic with NoC size, and not guaranteed to grow automatically with dimensionality. Its benefit depends on network configuration, traffic pressure, buffer size, and the position of the operating point relative to the knee.

The negative finding is also valuable:
Stage 1 does not prove that hand-tuned DP universally scales better than bufferlevel. Instead, it establishes DP and bufferlevel as characterised, valid baselines for later stages.

Stage 1 is therefore sufficiently concluded.
The recommended next step is Stage 2: DNN traffic trace generation.
Future RL stages can use these results as a baseline and investigate whether learned value functions can provide a stronger adaptive multi-hop metric than the hand-tuned DP cost field.
